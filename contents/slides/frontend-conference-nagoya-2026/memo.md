  - 出力者の AI と消費者の AI

2.  AIはWebの「出力者」になった

- チャット型の UI を多く扱うようになった。
- ストリーミングで逐次表示されるUIへの変化。これはフロントエンド設計の根本的な変化（確定したデータを表示するのではなく、生成途中のデータをリアルタイムに扱う）
- Mcp Apps https://modelcontextprotocol.io/extensions/apps/overview の登場、AIがテキストだけでなくインタラクティブなUIコンポーネントを返せるようになった。
  - 設計思想としては https://openai.com/ja-JP/index/introducing-apps-in-chatgpt/ も大事
  - ここで MCP Apps の実装について触れる
- AIが生成したコンテンツであることを示すメタデータ ai-disclosure属性の提案 https://github.com/dweekly/ai-content-disclosure
- ここでの問い：UIが“入口”ではなく“補助”になるという変化。呼び出される部品という考え。Webを「人間が操作するインターフェース」から
  「AIがオーケストレーションする実行基盤」へ変える。しかし、インタラクションと実行体験を設計する人というフロントエンドエンジニアの本質的な役割は変化しない

3.  AIはWebの「消費者」にもなった

- 今までもクローラはあったが、AIは内容を理解し、判断し、行動するという点で違う
- コンテキスト効率がよいマークダウンが注目され始めた。ウェブサイドがマークダウンバージョンを配信したり、claudeflare がエージェント向けにマークダウンを返す機能を導入 https://blog.cloudflare.com/ja-jp/markdown-for-agents/
- AI がウェブアプリをツールとして使う WebMCP https://github.com/webmachinelearning/webmcp が登場
  - ここで web mcp の実装にも触れる
- これまでのAPIとの違い、Webアプリが「人間のUI」と「AIのツールインターフェース」の二面性を持つようになること
- ここでの問い：単に AI に最適化されたコンテンツを返すのではなく、人間と AI が協業する方針に web が進んでいる、単に ai にツールを使わせるのであれば、web mcp を介する必要はない。web mcp はユーザーとエージェントが同じユーザーインターフェースでやり取りする。既存の web の資産を再利用できる 4.全体を通してのテーマ tool
  The Web is for everyone」の再解釈
  - everyoneにAIが含まれた — Tim Berners-Leeの理念は変わっていない。「everyone」の範 tool セシビリティの延長とも言える
  - すべての人に web というインターフェイスを介して体験を与える我々の役割は変わらないはず。これまで培ったHTML・CSS・JSの知識は無駄にならない。
