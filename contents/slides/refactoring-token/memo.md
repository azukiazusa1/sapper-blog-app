# ボーイスカウトルールでメモリやスキルを改善しよう


- プロジェクトの AGENTS.md, CLAUDE.md といったメモリやスキルの質はコーディングエージェントがより効率的に目的のタスクを達成するために重要
- プログラミングにおけるボーイスカウトルールとは
  - 「自分が通った場所を、来たときよりも少しでも良くして帰る」という考え方

- 従来はコードベースの改善というと、コードのリファクタリングやドキュメントの改善が中心だったが、エージェントのメモリやスキルも同じルールが適用できると考える
- 手を入れるべきポイントは「エージェントがタスクを進めている中で迷っていた個所はないか？」
- コーディングエージェントのハーネスがまだ整備されたいない初期の段階では、完全にエージェントを自律して動かすのではなく、エージェントがタスクを進めている様子を観察するとよい。ハーネスが整備され始めたら自律的に動かし始めて良い
  - マネジメントにおけるマイクロマネジメント → マクロマネジメントのように、最初は細かく観察して、徐々に任せていくというのに近い
- 迷っていた個所があれば、そこを改善することで次回以降のタスク遂行がよりスムーズになる。また同じチームのメンバーも同じタスクをする時に迷う可能性があるので、改善することでチーム全体の生産性向上にもつながる
- プロジェクト特有の暗黙知のような仕様やルールがあれば、CLAUDE.md や AGENTS.md に追記しておくことで
  - 例えば、
- 典型的に繰り返しの作業があれば、スキルにして手順を明文化しておくことで、次回以降の作業がスムーズになる
  - 私のプロジェクトではフロントエンドから　API を呼び出すとき、apiClient → service → hooks というレイヤー構造になっている
  - api の追加前にこのプロジェクトがどのような構造になっているか毎回調査する作業が必要になっていた
  - それぞれのレイヤーでテストを書くルールが守られていなかったり、service レイヤーの追加時に必要な作業を行わずに失敗することがなんどかあった
  - api の追加手順や検証手順をスキルとして明文化したことで、すぐに作業に取りかかれたり、失敗することが少なくなった
- メモリやスキルは単に追加していくだけでなく、定期的に見直して改善することも重要
  - 例えば、スキルの手順が古くなっていたり、より効率的な方法が見つかった場合は更新する
  - CLAUDE.md やスキルに既に知られている言語規約や一般的すぎる内容は書かない
    - プロジェクト特有のルールや暗黙知のような内容を中心に書く
    - 例えば PR を作成するスキルそのものは確かに使えるが、そのようなスキルは個人単位でインストールしていることも多く、重複してコンテキストを圧迫しやすい。
  - 実際に AI 時代の初期に追加された汎用的なスキルなどは見直して削除した
  - CLAUDE.md に日本語で会話してくださいなどは書かない
