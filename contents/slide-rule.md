# 技術発表スライド生成ルール

## 目的

このルールは、コーディングエージェントが Markdown 形式で技術発表スライドを作成・編集する際の基準を定めるものです。

スライドは、発表者の説明を補助し、聴衆が短時間で要点を理解できるように設計してください。

---

## 基本原則

### 1. 1スライドにつき1トピック

各スライドでは、1つの論点、結論、概念、処理、または比較だけを扱ってください。

複数の論点を含む場合は、スライドを分割してください。

各スライドについて、伝えたい内容を1文で説明できる状態にしてください。

---

### 2. タイトルには結論を書く

タイトルには、単なる分類名ではなく、そのスライドの主張や結論を書いてください。

避ける例:

```markdown
# ベンチマーク結果
```

推奨例:

```markdown
# キャッシュ導入により応答時間を40%短縮
```

ただし、章の区切りを示すセクションスライドでは、短い分類名を使用して構いません。

---

### 3. 投影を前提とした文字量にする

スライドは会場のスクリーンに投影されます。

後方の聴衆から読めるように、文字を小さくしなければ収まらない情報量を避けてください。

文字サイズの目安:

* タイトル: 32pt以上
* 見出し: 28pt以上
* 本文: 24pt以上
* 注釈、出典: 18pt以上
* コード: 20pt以上を推奨

Markdown上で文字サイズを直接指定しない場合も、このサイズで表示できる情報量に抑えてください。

---

### 4. 情報を詰め込まない

本文は、原則として3〜5項目以内にしてください。

1項目は1〜2行に収めてください。

長い段落や説明文は使用せず、短いフレーズ、図、表、コード、数値で表現してください。

内容が収まらない場合は、文字を小さくせず、スライドを分割してください。

---

## Markdown構造

### 5. 箇条書きの階層を深くしない

箇条書きは、原則として1階層までにしてください。

避ける例:

```markdown
- API
  - 認証
    - OAuth
      - PKCE
```

推奨例:

```markdown
- 認証方式: OAuth 2.0
- クライアント: Authorization Code + PKCE
- トークン管理: サーバー側で実施
```

階層が必要な場合は、図、表、または別スライドに変換してください。

---

## 配色と装飾

### 6. 配色は最低限にする

使用する色は、原則として以下の3種類までにしてください。

* 背景色
* 基本文字色
* 強調色

追加の色を使う場合は、情報上の明確な意味を持たせてください。

装飾だけを目的とした色の追加は禁止します。

---

### 9. 同じ意味には同じ表現を使う

色、アイコン、線、太字などの表現は、スライド全体で一貫させてください。

例:

* 赤: エラーまたは問題
* 緑: 成功または改善
* 青: 通常の強調

同じ色を異なる意味で使用しないでください。

---

### 10. 色だけに依存しない

状態や差分を色だけで示さないでください。

ラベル、記号、線種、太字、形状などを併用してください。

避ける例:

```text
赤い線が旧方式、緑の線が新方式
```

推奨例:

```text
旧方式: 赤色の破線
新方式: 緑色の実線
```

---

### 11. 装飾を増やしすぎない

以下は、必要な場合にのみ使用してください。

* 枠線
* 影
* グラデーション
* 背景画像
* アイコン
* アニメーション
* 絵文字

装飾よりも、情報の優先順位と余白を重視してください。

---

## コード

### 12. コードは必要な箇所だけ掲載する

コード全体を掲載せず、説明に必要な部分だけを抜粋してください。

目安:

* 1スライドにつき10〜15行以内
* 1行は80文字以内を推奨
* 横スクロールが必要なコードは禁止
* 不要なimport、設定、例外処理は省略可能
* 省略箇所はコメントで明示

例:

```python
def get_user(user_id: str) -> User:
    cached = cache.get(user_id)
    if cached is not None:
        return cached

    user = repository.find(user_id)
    cache.set(user_id, user)
    return user
```

---

### 13. コードの重要箇所を明示する

説明対象となる行を、太字、ハイライト、行番号、コメント、差分形式などで明示してください。

使用するスライドツールが行ハイライトに対応している場合は、その機能を優先してください。

コード全体を複数色で装飾しないでください。

---

### 14. 差分はdiff形式を優先する

変更前後を説明する場合は、可能な限りdiff形式を使用してください。

```diff
- result = fetch_from_database(user_id)
+ result = cache.get(user_id) ?? fetch_from_database(user_id)
```

変更箇所が多い場合は、複数スライドに分割してください。

---

### 15. コードと説明を重複させない

コードから明らかな内容を、そのまま箇条書きで繰り返さないでください。

説明では、以下を補足してください。

* なぜこの実装にしたか
* どの制約に対応しているか
* 何が改善されたか
* どの部分に注意が必要か

---

## 図表

### 16. 構造や処理は図で表現する

以下の内容は、文章よりも図を優先してください。

* システム構成
* データフロー
* シーケンス
* 状態遷移
* 依存関係
* 処理手順
* 比較
* 時系列

図は、発表内容の理解に必要な要素だけで構成してください。

---

### 17. Mermaid図は単純に保つ

Mermaidなどのコード生成図を使う場合は、ノード数を抑えてください。

目安:

* 1図につき5〜9ノード程度
* ノード内の文章は短くする
* 線の交差を減らす
* 方向は原則として左から右、または上から下
* 1つの図で複数の抽象度を混在させない

複雑な図は、全体図と詳細図に分割してください。

---

### 18. 表は比較に限定する

表は、複数の項目を同じ基準で比較する場合に使用してください。

表の列数は、原則として4列以内にしてください。

各セルに長文を入れないでください。

避ける例:

```markdown
| 項目 | 詳細な説明 | 補足説明 | 制約 | 注意事項 | 備考 |
```

推奨例:

```markdown
| 方式 | レイテンシ | 実装コスト |
|---|---:|---:|
| DB直接参照 | 120 ms | 低 |
| キャッシュ利用 | 35 ms | 中 |
```

---

### 19. グラフには結論を添える

グラフを配置するだけで終わらせず、読み取るべき結論をタイトルまたは短い注記で示してください。

必ず以下を明記してください。

* 軸名
* 単位
* 対象期間
* 比較条件
* データの出典

3Dグラフは使用しないでください。

---

## 画像とスクリーンショット

### 20. スクリーンショットは必要部分だけ切り出す

画面全体のスクリーンショットをそのまま使用しないでください。

以下を実施してください。

* 不要な領域を切り取る
* 個人情報や機密情報を除去する
* 注目箇所を枠や矢印で示す
* 文字が読める大きさに拡大する
* 複数の注目箇所がある場合は分割する

---

### 21. 画像の比率を崩さない

画像を拡大・縮小する場合は、縦横比を維持してください。

低解像度の画像を無理に拡大しないでください。

画像内の文字が投影時に読めない場合は、画像を使わず、内容を再構成してください。

---

## 発表構成

### 22. 発表全体に一貫した流れを持たせる

技術発表は、原則として以下の流れで構成してください。

1. 背景
2. 課題
3. 目的
4. アプローチ
5. 実装
6. 結果
7. 考察
8. まとめ

発表内容に応じて変更して構いませんが、各章の役割を明確にしてください。

---

### 23. 冒頭で前提とゴールを示す

発表の序盤で、以下を明示してください。

* 対象となるシステムや技術
* 聴衆に必要な前提知識
* 解決する課題
* 発表終了時に理解してほしいこと

前提知識が多い場合は、本編で全て説明せず、必要最低限に絞ってください。

---

### 24. 最後に要点を再提示する

まとめスライドでは、聴衆に持ち帰ってほしい内容を2〜3点に絞ってください。

新しい論点を追加しないでください。

推奨形式:

```markdown
# まとめ

- キャッシュによりP95レイテンシを70%削減
- 無効化戦略にはイベント駆動方式を採用
- 次の課題は障害時のフォールバック改善
```

---

### 25. 詳細情報は補足スライドに移す

以下の情報は、必要に応じて補足スライドに配置してください。

* 詳細な設定値
* 全ベンチマーク結果
* 長いコード
* 詳細なアーキテクチャ
* 代替案の比較
* 用語集
* 想定質問への回答
* 参考資料

本編の理解に必須でない情報を、本編へ詰め込まないでください。

---

## 数値と根拠

### 26. 数値には条件を付ける

性能、削減率、改善率などの数値には、測定条件を明記してください。

悪い例:

```text
処理速度が3倍になった
```

良い例:

```text
同時接続100、データ件数100万件の条件で、
P95レイテンシが120msから40msに改善
```

---

### 27. 出典を明記する

外部の数値、図、文章、画像、調査結果を使用する場合は、出典を記載してください。

出典には、可能な範囲で以下を含めてください。

* 資料名
* 著者または組織名
* 公開年
* URL
* 参照日

長いURLは、脚注または参考資料スライドに配置してください。

---

### 28. 推測と事実を区別する

未検証の内容を、確定事項として記載しないでください。

以下のように区別してください。

* 測定結果
* 観測事実
* 仮説
* 推測
* 今後の検証事項

---

## エージェントの生成・編集ルール

### 29. 情報が多い場合は自動的に分割する

1枚に収まらない場合、文字サイズを下げるのではなく、複数スライドへ分割してください。

分割時は、タイトルに連番だけを付けるのではなく、各スライドの論点が分かるタイトルを付けてください。

避ける例:

```markdown
# アーキテクチャ 1
# アーキテクチャ 2
```

推奨例:

```markdown
# 書き込み処理はイベント経由で非同期化
# 読み取り処理はキャッシュを優先
```

---

### 30. 不足情報を推測で補完しない

測定値、固有名詞、URL、日付、構成、実装内容などが不足している場合は、事実として捏造しないでください。

仮の値を使う場合は、明確にプレースホルダーとして示してください。

例:

```markdown
- P95レイテンシ: `{{P95_LATENCY}} ms`
- 測定期間: `{{MEASUREMENT_PERIOD}}`
```

---

### 31. 元の技術的意味を変更しない

文章を短くする際も、以下を変更しないでください。

* 前提条件
* 因果関係
* 数値
* 単位
* 制約
* 例外
* 不確実性
* 技術用語の意味

正確性と簡潔さが競合する場合は、正確性を優先してください。

---

### 32. 発表者が口頭で説明する内容を残す

スライドだけで全てを説明しようとしないでください。

スライドには、以下を中心に残してください。

* 結論
* キーワード
* 数値
* 図
* 比較
* コード
* 判断根拠

詳細な背景説明や接続表現は、発表者ノートへ移してください。

---

### 33. 発表者ノートとスライド本文を分離する

使用するスライドツールが発表者ノートに対応している場合は、本文とノートを明確に分離してください。

発表者ノートには、以下を記載できます。

* 詳細な説明
* 次のスライドへの接続
* 補足情報
* 想定質問
* 発表時の注意
* デモ手順

ノートの内容をスライド本文へそのまま掲載しないでください。

---

### 34. スライドツール固有の記法を優先する

Marp、Slidev、reveal.jsなど、対象ツールが指定されている場合は、そのツールの正式な記法に従ってください。

異なるツールの記法を混在させないでください。

対象ツールが不明な場合は、標準的なMarkdownを使用し、ツール固有機能への依存を避けてください。

---

### 35. 生成後に自己レビューする

生成または修正後は、以下を確認してください。

* 各スライドが1トピックになっているか
* タイトルが論点や結論を表しているか
* 本文が多すぎないか
* コードが長すぎないか
* 図が複雑すぎないか
* 色を使いすぎていないか
* 数値に単位と条件があるか
* 出典が必要な箇所に記載されているか
* 同じ内容を複数スライドで重複していないか
* 全体の流れが自然か
* まとめが発表内容と一致しているか

違反がある場合は、出力前に修正してください。

---

## 禁止事項

以下は禁止します。

* 文字を小さくして情報を詰め込む
* 1枚に複数の独立した論点を載せる
* 長い段落をそのまま掲載する
* コードファイル全体を掲載する
* 横スクロールが必要なコードを載せる
* 過度な色、装飾、アニメーションを使う
* 低解像度の画像を拡大する
* 出典のない外部データを事実として掲載する
* 未確認の数値や実装を捏造する
* 色だけで状態や差分を表現する
* 本編に補足情報を詰め込む
* スライドツール固有の記法を混在させる

---

## 出力前チェックリスト

* [ ] 1スライドにつき1トピックである
* [ ] タイトルから主張または結論が分かる
* [ ] 本文は3〜5項目程度に収まっている
* [ ] 長い文章を掲載していない
* [ ] 投影時に読める文字量である
* [ ] 使用する色を最低限に抑えている
* [ ] 色だけに依存した表現がない
* [ ] 箇条書きの階層が深くない
* [ ] コードは10〜15行程度に収まっている
* [ ] コードの注目箇所が明確である
* [ ] 図表の要素数が多すぎない
* [ ] グラフに軸名、単位、条件がある
* [ ] 数値に測定条件がある
* [ ] 外部情報に出典がある
* [ ] 未確認情報を事実として補完していない
* [ ] 発表者ノートと本文が分離されている
* [ ] スライド全体の構成が一貫している
* [ ] 最後に要点をまとめている
* [ ] 詳細情報を補足スライドへ分離している
